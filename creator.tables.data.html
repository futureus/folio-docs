Data Entry

The most common form of data you will be entering into your table is plain text. To enter text, simply click in the relevant cell and type.

To set text in the section header for a column, tap on the header for the appropriate column to bring up the column settings menu ( "Column, row and section header settings menus"), and enter the text in the Title field.

You may want your table cells to display images rather than text. To achieve this, place your images into your issue folder and type the image filename into the appropriate cell, surrounded by square brackets, e.g. [myImage.png]. The image will be aligned to the left of the cell horizontally, and centre-aligned vertically. Any overspill will be clipped to the cell size. Note that images are not supported in section headers - only table cells.

When a table is initially displayed, the data is arranged in the same order in which it was entered by you. When the user taps on the section header for a column, the data in that section and column will be sorted alphabetically. Tapping again will reverse the sort.

Cells which contain image data will be included in the sort operation, using their filenames (including the square brackets) as the data by which to sort. With careful file naming, you can therefore use this feature to have a column containing e.g. star rating images which then sort correctly.

The last type of data which you can enter into cells is sortable dates. If you enter a date value as plain text, it will not necessarily sort in logical date order - alphabetical sorting will be applied to it. To achieve correct date sorting, a special formatting has been introduced: <yyyy.mm.dd.hh.mm.ss>. This formatting (including the angle brackets) will be detected and resolved into a date by which to sort. It will then be displayed in a user-friendly manner (see "Date-formatted entries' display and sort").

The accepted forms of date entry are:

  • <yyyy.mm.dd.hh.mm.ss> - note that you only have to fill this out up to the desired point, e.g. <2012.02> is fine.
  • <Out now>
  • <TBC>
  • <N/A>
  • <Q1>, <Q2> etc - Q represents "quarter".
  • <yyyy.Q1>, <yyyy.Q2> etc.
  • <Spring>, <Summer> etc - both "Autumn" and "Fall" are supported.
  • <yyyy.Spring>, <yyyy.Summer> etc.

The entries that involve words are not case sensitive, so <OUT NOW> is valid, as is <out now> and so on. When it comes to display the value, your casing will be respected.

When the sort operation is performed, the <Out now>, <TBC> and <N/A> entries will be interpreted as a date in the distant past, so that they appear logically in the sequence, e.g. Out now -> in 1 month -> in 2 months etc.

For the sort operation, the season and quarter entries are interpreted as the last day of that season or quarter - which evaluates to the following dates:

  • Q1 - March 31st.
  • Q2 - June 30th.
  • Q3 - September 30th.
  • Q4 - December 31st.
  • Spring - June 20th.
  • Summer - Sept 21st.
  • Autumn / Fall - December 20th.
  • Winter - March 20th the following year (i.e. <2012.Winter> -> March 20th 2013).

Where no years are supplied, the current year at the time the user is viewing it is assumed.

When it comes to displaying your date-formatted entries in the table, the angle brackets will be omitted; your casing will be used, and the following formatting applied:

  • <2012.05.29.10.01.30> - 10:01:30 29 May 2012.
  • <2012.05.29> -> 29 May 2012
  • <Out Now> -> Out Now
  • <TBC> -> TBC
  • <N/A> -> N/A
  • <Q1> -> Q1
  • <2023.Q2> -> Q2 2023
  • <Winter> -> Winter
  • <2023.Winter> -> Winter 2023
